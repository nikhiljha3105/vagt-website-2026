# VAGT Security Services — Platform Audit Report
**Date:** 2026-03-15
**Scope:** Full-stack security audit of VAGT Security Services platform
**Stack:** Firebase Hosting, Cloud Functions (Node 20 + Express), Firestore, Firebase Auth

---

## Executive Summary

The VAGT platform demonstrates solid foundational architecture with careful attention to authentication, rate-limiting, and Firestore security rules. However, **critical gaps exist in frontend data access patterns, CSP policy enforcement, input validation consistency, and error handling**. The platform is **not production-ready for Indian security services compliance** — missing PSARA Act audit trails, data retention policies, and labour law integrations.

**Critical Issues:** 5
**High Issues:** 18
**Medium Issues:** 14
**Low Issues:** 8

## 1. SECURITY

### 1.1 Authentication & Authorization

#### CRITICAL: Hardcoded Setup Passphrase Exposed in Code
- **File:** `firebase/functions/src/index.js`, line 165
- **Issue:** `SETUP_PASSPHRASE = 'VAGT-SETUP-2026'` is plaintext in code. If committed to public repo or backup, this enables unauthorized first-admin setup.
- **Impact:** Single-factor admin bootstrap — no secondary verification.
- **Fix:** Move to Firebase environment config or Cloud Secret Manager. Require backend verification via logged-in user credentials.
- **Effort:** S

#### HIGH: Frontend Direct Firestore Writes (36+ instances)
- **Files:** All portal HTML files (`admin-portal.html`, `client-portal.html`, `employee-*.html`)
- **Issue:** Pages use Firebase Client SDK to directly write to Firestore collections: `db.collection(...).add()`, `.update()`, `.delete()`. Examples:
  - `admin-portal.html`: reads `leave_requests`, `activity_log` directly
  - `client-portal.html`: reads `sites`, `attendance_logs`, `complaints` directly
  - `employee-schedule.html`: reads `shifts` directly
- **Impact:** Bypasses API middleware auth checks, rate-limiting, and audit logging. Violates principle of least privilege. Could enable data exfiltration if auth token is compromised.
- **Risk:** Guard can craft custom Firestore queries (no query bounds), read/write data for other users if rule is permissive.
- **Fix:** All Firestore writes must go through API routes (`/api/...`). Firestore rules should only allow read-only from frontend (for specific, scoped collections). Example:
  ```
  match /attendance_logs/{logId} {
    allow read: if isAnyEmployee() && resource.data.employee_uid == request.auth.uid;
    allow write: if false;  // ← all writes via API only
  }
  ```
- **Effort:** L (requires refactoring many pages)

#### HIGH: Missing Input Validation on Multiple Routes
- **File:** `firebase/functions/src/routes/admin.js`, lines 426–445 (schedule POST)
- **Issue:** `POST /api/admin/schedule` accepts `employee_id`, `site_id`, `date`, `shift_type` but doesn't validate formats:
  - `date` — not validated as `YYYY-MM-DD`; accepts any string
  - `employee_id` — not checked if employee exists or is active
  - `site_id` — not validated; could be random UUID
  - `shift_type` — accepts any string (no enum check)
- **Impact:** Invalid data stored in Firestore; schedule page renders broken state.
- **Similar Issues:**
  - `POST /api/leave/apply` (employee.js:268) — no date format validation
  - `POST /api/guest/entry` (guest.js:35) — `visitor_type` validated but `site_id` is not
  - `POST /api/patrol/checkpoint` (patrol.js:36) — `nfc_tag_id` not validated for format
- **Fix:** Add strict validation library (joi/zod) or manual regex checks before db writes.
- **Effort:** M

#### HIGH: No Logout on Client-Side Auth State Loss
- **Files:** All portal pages
- **Issue:** Firestore rules and API middleware require valid Firebase token. But frontend doesn't handle token expiry gracefully:
  - Token expires after 1 hour (Firebase default)
  - Frontend has `onAuthStateChanged` listener, but doesn't refresh token for API calls
  - API calls with expired token get 401; UI shows error but user stays on page
- **Impact:** Stale session; user thinks they're still logged in.
- **Fix:** Wrap all fetch calls in a token-refresh helper:
  ```javascript
  const getAuthHeader = async () => {
    const token = await auth.currentUser.getIdToken();
    return { Authorization: `Bearer ${token}` };
  };
  ```
- **Effort:** M

#### HIGH: Firestore Rules Allow Read-All for Sites Collection
- **File:** `firebase/firestore.rules`, line 144
- **Issue:**
  ```
  match /sites/{siteId} {
    allow read: if isAnyEmployee() || isAnyClient() || isAdmin();
    allow write: if isAdmin();
  }
  ```
  Any logged-in employee can read ALL sites (not just their assigned sites). Violates data compartmentalization.
- **Impact:** Guard can enumerate all clients' sites; client can see all sites (not just their own).
- **Fix:** Restrict reads to user-specific sites:
  ```
  allow read: if isAnyEmployee() &&
    resource.data.employee_ids.contains(request.auth.uid);
  allow read: if isAnyClient() && resource.data.client_uid == request.auth.uid;
  ```
- **Effort:** M

#### MEDIUM: Missing Rate-Limit on `/api/admin/schedule DELETE`
- **File:** `firebase/functions/src/routes/admin.js`, line 478
- **Issue:** `DELETE /api/admin/schedule/:id` has no `actionLimiter` middleware. Admin can repeatedly delete shifts without rate-limit.
- **Impact:** Accidental or malicious rapid-fire deletion of shifts.
- **Fix:** Add `actionLimiter` to the route.
- **Effort:** S

#### MEDIUM: OTP Never Expires After Initial Verification
- **File:** `firebase/functions/src/routes/auth.js`, line 302
- **Issue:** In `POST /api/auth/employee/verify-otp`, after successful verification, the `otp` field is not cleared. If a duplicate request arrives (same registration_token, wrong OTP), it fails correctly. But if attacker knows old OTP, they can resend registration and try old OTP again.
- **Impact:** Low (OTP is time-limited, only valid 15 min), but expired OTP should be cleared.
- **Fix:** Update `await docRef.update({ otp: null, ... })` after successful verify.
- **Effort:** S

#### MEDIUM: Admin Password Reset Link Never Sent
- **File:** `firebase/functions/src/routes/admin.js`, lines 218–222
- **Issue:** Commented-out SMS send on admin approval. Guard can't set password without SMS integration. Password reset link generation requires email setup (Firebase requires SMTP or 3rd party).
- **Impact:** Blocked UX — new guards can't set password on day 1.
- **Fix:** Wire MSG91 SMS or use Firebase email link (if available).
- **Effort:** M

### 1.2 Firestore Security Rules — Gaps

#### MEDIUM: activity_log Write Restriction Not Enforced
- **File:** `firebase/firestore.rules`, line 158
- **Issue:**
  ```
  allow write: if false; // Only server-side admin SDK writes
  ```
  This is a good intent, but Firestore rules don't actually prevent the Admin SDK from writing (admin SDK has full access). The `if false` is documentation, not enforcement. Frontend will get 403 if it tries to write, which is correct, but the comment is misleading.
- **Impact:** Clarity — developers might think this prevents server-side writes (it doesn't).
- **Fix:** Update comment to clarify: `// Server-side writes via admin SDK bypass rules. Client writes always denied.`
- **Effort:** S

#### MEDIUM: Incidents Collection Missing Cross-Client Compartmentalization
- **File:** `firebase/firestore.rules`, line 97–108
- **Issue:**
  ```
  allow read: if isAnyEmployee() && resource.data.employee_uid == request.auth.uid;
  allow read: if isAdmin();
  allow write: if isAdmin();
  ```
  Employees can only read their own incidents (good). But if a client portal is added, there's no rule for client to read incidents at their sites. Incidents need `site_id` field and cross-checks.
- **Impact:** Future client portal can't show incidents (missing rule).
- **Fix:** Add role-based incident rules before client portal launches.
- **Effort:** M

### 1.3 Content Security Policy (CSP)

#### HIGH: Overly Permissive CSP — `unsafe-inline` Everywhere
- **Files:** Multiple HTML files and `firebase.json`
- **Issue:** CSP headers allow `script-src 'self' 'unsafe-inline'` and `style-src 'self' 'unsafe-inline'`:
  - `index.html`: allows inline scripts
  - `firebase.json`: allows inline scripts and styles on all pages
  - `admin-portal.html`, `employee-portal.html`, etc.: each redefine CSP with inline
- **Impact:** XSS vulnerability — inline event handlers (e.g., `onclick="..."`) are allowed. If attacker injects HTML, inline JS executes.
- **Risk Example:** Reflected XSS in error message → `<h1 onclick="stealToken()">...</h1>` would execute.
- **Fix:** Remove `unsafe-inline` and move all inline scripts to external files; use nonces for necessary inline styles:
  ```
  script-src 'self' 'nonce-${random}' https://www.gstatic.com;
  style-src 'self' 'nonce-${random}' https://fonts.googleapis.com;
  ```
- **Effort:** L (low-hanging fruit)

#### MEDIUM: CSP Doesn't Include Firebase JS SDK Origins
- **File:** `firebase.json`
- **Issue:** CSP `script-src` doesn't explicitly list Firebase SDK domains. It allows `https://www.gstatic.com https://apis.google.com`, but Firebase client SDK might load from `https://www.googletagmanager.com` or other origins.
- **Impact:** Implicit trust in Google; if Google CDN is compromised, malicious script can run.
- **Fix:** Audit all Firebase SDK dependencies and explicitly list in CSP.
- **Effort:** S

### 1.4 API Security

#### HIGH: No Input Validation for Numeric Fields
- **File:** `firebase/functions/src/routes/auth.js`, line 386–421 (keycode-login)
- **Issue:** `latitude`, `longitude`, `accuracy` are accepted as `typeof number` but not range-checked:
  - `latitude` — not validated: -90 to 90
  - `longitude` — not validated: -180 to 180
  - `accuracy` — not validated: should be > 0 (meters)
- **Impact:** Garbage GPS data stored in sign_in_events.
- **Fix:** Add range checks:
  ```javascript
  if (typeof latitude !== 'number' || latitude < -90 || latitude > 90)
    return res.status(400).json({ message: 'Invalid latitude.' });
  ```
- **Effort:** S

#### HIGH: Missing Deletion Idempotency Check
- **File:** `firebase/functions/src/routes/admin.js`, line 478
- **Issue:** `DELETE /api/admin/schedule/:id` doesn't check if shift exists before deletion. If called twice, 2nd call silently succeeds (Firestore `delete()` is idempotent).
- **Impact:** No clear error feedback. Activity log doesn't record double-delete.
- **Fix:** Check `exists` before delete, log accordingly.
- **Effort:** S

#### MEDIUM: Unbounded Queries on activity_log
- **File:** `firebase/functions/src/routes/admin.js`, line 131
- **Issue:** `GET /api/admin/activity` returns 50 most recent logs. But `activity_log` is unbounded — no TTL, no archival. After 2 years, this will have millions of docs; query slows down.
- **Impact:** Disk usage grows indefinitely. Firestore index size explodes.
- **Fix:** Add TTL via scheduled function: delete docs older than 90 days. Or implement pagination with `startAfter()`.
- **Effort:** M

#### MEDIUM: No Validation on Leave Date Ranges
- **File:** `firebase/functions/src/routes/employee.js`, line 268
- **Issue:** `POST /api/leave/apply` accepts `from_date` and `to_date` but doesn't validate:
  - Format (should be YYYY-MM-DD)
  - Order (`from_date` <= `to_date`)
  - Range (max leave at once, e.g., 30 days)
- **Impact:** Invalid dates stored; admin confused.
- **Fix:** Add date parsing and validation.
- **Effort:** S

## 3. CODE QUALITY

### 3.1 Duplicate Code Patterns

#### MEDIUM: Repeated Auth Guard Pattern in All Routes
- **Files:** `routes/auth.js`, `routes/employee.js`, `routes/client.js`, `routes/admin.js`, `routes/guest.js`, `routes/patrol.js`
- **Issue:** Every route file re-defines `const guard = [requireAuth, requireRole]`. Same pattern repeated 6 times.
- **Impact:** DRY violation; harder to update auth flow globally.
- **Fix:** Export guard factory from index.js:
  ```javascript
  const createGuard = (roles) => [
    requireAuth,
    (req, res, next) => {
      if (!roles.includes(req.user.role)) return res.status(403).json({...});
      next();
    }
  ];
  ```
- **Effort:** S

#### MEDIUM: Duplicate Date Formatting Logic
- **Files:** `routes/employee.js` (line 320), `routes/admin.js`, `routes/guest.js`
- **Issue:** Multiple places convert period 'YYYY-MM' to month label:
  ```javascript
  new Date(year, parseInt(month, 10) - 1).toLocaleString('en-IN', { month: 'long' })
  ```
  Repeated 5+ times.
- **Impact:** Maintenance burden; locale changes require updates in multiple files.
- **Fix:** Create utility function `formatMonthLabel(period)`.
- **Effort:** S

### 3.2 Error Handling Gaps

#### HIGH: Unhandled Promise Rejection in Guest Entry
- **File:** `routes/guest.js`, line 78
- **Issue:**
  ```javascript
  const ref = await db.collection('guest_logs').add({...});
  const qrDataUrl = await QRCode.toDataURL(token, {...});
  return res.status(201).json({ ..., qr_data_url: qrDataUrl });
  ```
  If `QRCode.toDataURL()` fails (out of memory, invalid args), no catch block. Exception bubbles to global handler, response is already sent (201).
- **Impact:** Client receives 201 + incomplete JSON; QR slip can't be printed.
- **Fix:** Wrap QR generation in try/catch; if it fails, return 201 without QR (generate on next request).
- **Effort:** S

#### MEDIUM: Missing Error Logging Context
- **Files:** All route handlers
- **Issue:** Errors are logged as `console.error('route error:', err)`, but don't include request context (user ID, request body, timestamp in ISO format).
- **Impact:** Hard to debug in production; can't trace which user triggered error.
- **Fix:** Log structured errors:
  ```javascript
  console.error(JSON.stringify({
    timestamp: new Date().toISOString(),
    endpoint: req.path,
    userId: req.user?.uid,
    error: err.message,
  }));
  ```
- **Effort:** M

#### MEDIUM: No Timeout on Firestore Queries
- **Files:** All route handlers
- **Issue:** Firestore queries don't have explicit timeout. If Cloud Firestore region is slow or down, request hangs until Cloud Function timeout (9 minutes for HTTP triggers).
- **Impact:** Slow user experience; can't retry quickly.
- **Fix:** Set query timeout in application code (Node.js native `AbortSignal` + timeout).
- **Effort:** M

## 4. PERFORMANCE

### 4.1 Firestore Query Optimization

#### HIGH: Missing Composite Indexes
- **Files:** `firebase/functions/src/routes/admin.js`, `client.js`, `patrol.js`
- **Issue:** Queries with multiple where clauses and orderBy will need composite indexes:
  - `admin.js:521`: `where('client_uid', ...).where('status', 'in', ...).` — needs index
  - `patrol.js:124`: `where('guard_uid', ...).where('scanned_at', '>=', ...).where('scanned_at', '<=', ...).orderBy('scanned_at')` — needs index
- **Impact:** Firestore will refuse to execute without indexes; users get error.
- **Fix:** Run queries in development; Firestore UI will prompt to create indexes. Commit indexes to `firebase/firestore.indexes.json`.
- **Effort:** S (one-time setup)

#### HIGH: Unbounded Query Without Pagination
- **File:** `routes/admin.js`, line 96
- **Issue:** `GET /overview` runs:
  ```javascript
  db.collection('employees').limit(500).get()
  ```
  If 300 employees and 100 new registrations, this reads 600 docs. Each doc access costs 1 Firestore read. At scale (500+ employees), this burns through quota.
- **Impact:** Read cost grows linearly; no pagination.
- **Fix:** Add pagination with cursor (`startAfter(lastDoc)`).
- **Effort:** M

#### MEDIUM: N+1 Query Pattern in Several Places
- **File:** `routes/admin.js`, line 268–282 (pending-leaves)
- **Issue:**
  ```javascript
  const snap = await db.collection('leave_requests').where('status', '==', 'pending').get();
  // Then for each doc:
  const empDocs = empRefs.length > 0 ? await db.getAll(...empRefs) : [];
  ```
  This is actually optimized (batch fetch with `db.getAll()`), which is good. But other routes don't use this pattern.
- **Example of bad pattern:** `admin.js:596` (active guards):
  ```javascript
  todayShifts.docs.map(s => ({ ... s.data().employee_name ... }))
  ```
  `employee_name` is denormalized (good), but if it wasn't, this would be N+1.
- **Fix:** Continue using `db.getAll()` for batch reads.
- **Effort:** S

## 5. INDUSTRY STANDARDS GAP

### 5.1 Indian Labour Law & PSARA Compliance

#### CRITICAL: No PSARA Compliance Audit Trail
- **Issue:** Private Security Agency Regulation Act (PSARA) requires:
  - Guard attendance verification (photo + timestamp)
  - Shift coverage proof (who was on duty, when, where)
  - Incident reporting with FIR reference
  - Guard background check record + renewal
- **Current Implementation:** Attendance logged as check-in/out timestamps. No photo verification. No background check tracker.
- **Gap:** VAGT cannot prove to government/clients that guards actually worked assigned shifts.
- **Fix:** Add:
  1. Shift photo verification (guard takes selfie with timestamp at start/end)
  2. Background check collection with renewal dates
  3. Export audit report for PSARA submission
- **Effort:** L (architectural change)

#### CRITICAL: No Data Retention Policy
- **Issue:** Personal data (phone, Aadhaar last 4, address) stored indefinitely. GDPR-like data protection laws in India (e.g., Karnataka) require deletion policy.
- **Impact:** Legal liability if data is breached; no retention justification.
- **Fix:** Implement TTL:
  - Delete inactive employee records after 3 years
  - Delete incident logs after 7 years (compliant with Indian labour law)
  - Delete guest logs after 90 days
- **Effort:** M

#### HIGH: Missing Payroll Audit Trail
- **Issue:** `POST /admin/payroll/run` generates payslips but:
  - No approval workflow (who authorized payroll run?)
  - No change log (what changed from last month?)
  - No reconciliation report
- **Impact:** If VAGT is audited, can't prove payroll was accurate.
- **Fix:** Add approval state; log who ran payroll and when.
- **Effort:** M

#### MEDIUM: Aadhaar Data Security Not Addressed
- **Issue:** Employee registration captures `aadhaar_last4`, which could escalate to full Aadhaar number. No data masking in UI.
- **Impact:** If Aadhaar is later stored in full, it's PII — must be encrypted at rest and in transit.
- **Fix:** Document that only last 4 are stored; never accept full Aadhaar; encrypt SSN/Aadhaar fields if added.
- **Effort:** S

### 5.2 Data Protection

#### HIGH: No Encryption at Rest Specified
- **Issue:** Firestore stores sensitive employee data (phone, address, Aadhaar last 4, incident reports). No field-level encryption configured.
- **Impact:** If Firestore is compromised, PII is readable.
- **Fix:** Enable Firestore data encryption (Firebase-managed keys by default are fine for MVP, but consider CMEK for production).
- **Effort:** S (configuration)

#### MEDIUM: No Access Control Audit Log
- **Issue:** Firestore rules check `request.auth.uid == resource.data.employee_uid`, but there's no log of who accessed what. If a guard downloads another guard's payslip, there's no audit trail.
- **Impact:** Can't detect unauthorized access.
- **Fix:** Log all data access to `access_logs` collection (add Firestore trigger or API middleware).
- **Effort:** M

## 6. PRIORITISED ACTION LIST

### CRITICAL (Do before launch)

1. **Remove Hardcoded Setup Passphrase**
   - Severity: CRITICAL
   - Location: `firebase/functions/src/index.js`, line 165
   - Description: `SETUP_PASSPHRASE = 'VAGT-SETUP-2026'` is plaintext. Move to Firebase environment config and require second-factor verification (logged-in admin + email confirm).
   - Recommended Fix: Use Firebase Secret Manager; require admin email OTP.
   - Effort: S

2. **Eliminate Direct Firestore Writes from Frontend**
   - Severity: CRITICAL
   - Location: All portal HTML pages (36+ instances)
   - Description: Pages use Firebase Client SDK to directly read/write Firestore. This bypasses API auth, rate-limiting, and logging. All data mutations must go through `/api/...` routes.
